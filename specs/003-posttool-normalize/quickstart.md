# Kata 3 — Quickstart

## What you'll build

A `PostToolUseHook` that interposes between a legacy data tool and the
model's conversation history. The hook persists the raw XML + arcane status
codes to an append-only audit log, then emits a minimal pydantic
`NormalizedPayload` — the only artifact the model ever sees. Running the
kata in baseline mode (hook off) reproduces the anti-pattern: raw legacy
markup dumped into context, saturating tokens and inviting misread status
codes. Running it in normalized mode (hook on) proves a ≥70% token
reduction with zero markup leakage.

## Prerequisites

- Python 3.11+
- `uv` or `pip`
- An Anthropic API key in `ANTHROPIC_API_KEY` — **only** needed for the live
  comparison run; default tests use recorded fixtures.

## Install

```bash
# From repo root
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"   # pyproject.toml declares: anthropic, pydantic, lxml, pytest, pytest-bdd
```

## Run the kata against recorded fixtures (default — no API key needed)

```bash
pytest tests/katas/003_posttool_normalize -v
```

You should see:
- the Gherkin `posttool_normalize.feature` scenarios pass (US1 / US2 / US3),
- `test_normalizer_unknown_code.py` pass (FR-003 / SC-003 — unknown code
  surfaces as the `unknown` marker with raw preserved),
- `test_hook_markup_leak.py` pass (FR-004 / SC-002 — no `<`, `>`, CDATA
  sentinels in any string field of the emitted payload),
- `test_audit_roundtrip.py` pass (FR-005 / SC-004 — SHA-256 of every
  fixture matches the decoded `raw_bytes_b64` in `audit.jsonl`),
- `test_token_reduction.py` pass (SC-001 — normalized tokens ≤ 30% of raw,
  averaged across the fixture corpus),
- `test_mapping_extension.py` pass (US3 — adding one entry to
  `STATUS_MAPPING` surfaces the mapped label with no other edits).

## Run the baseline vs. normalized comparison (live or fixture-driven)

```bash
# Fixture-driven (default — deterministic):
python -m katas.003_posttool_normalize.runner --fixture happy_path --compare

# Live API (opt-in):
LIVE_API=1 python -m katas.003_posttool_normalize.runner \
  --model claude-opus-4-7 \
  --prompt "Query the legacy registry for account 4471." \
  --compare
```

Artifacts produced per session:
- `runs/<session-id>/audit.jsonl` — one `AuditRecord` per intercepted
  response, raw payload bytes preserved.
- `runs/<session-id>/comparison.json` — `{raw_tokens, normalized_tokens,
  reduction_ratio, markup_leaks_baseline, markup_leaks_normalized}`.

## Verify the anti-pattern defense

```bash
# 1. Confirm zero markup leaks in normalized runs:
jq 'select(.parse_status != "empty") | .normalized_token_count as $n
    | .raw_token_count as $r | {ratio: ($n / $r)}' \
    runs/<session-id>/audit.jsonl

# 2. Confirm raw payloads are byte-for-byte recoverable:
jq -r '.raw_bytes_b64' runs/<session-id>/audit.jsonl \
  | while read B64; do echo "$B64" | base64 -d | sha256sum; done
```

## §Kata Completion Standards checklist

Per Constitution v1.3.0 §"Kata Completion Standards":

- [ ] `spec.md`, `plan.md`, `tasks.md`, and `.feature` acceptance file all
      exist for this kata. Plan done; tasks generated by `/iikit-05-tasks`;
      `.feature` generated by `/iikit-04-testify`.
- [ ] Acceptance scenarios cover the objective (US1) AND the anti-pattern
      (US2 baseline comparison).
- [ ] Evaluation harness asserts signal-level properties: pydantic schema
      conformance, `audit.jsonl` SHA-256 roundtrip, token-ratio threshold —
      never string-matching over model output.
- [ ] Anti-pattern test fails closed: swapping the hook for a no-op leaves
      raw legacy markup in the message appended to history, and
      `test_hook_markup_leak.py` goes red.
- [ ] Assertion-integrity hashes in `.specify/context.json` match the
      locked test set — produced by `/iikit-04-testify`, verified by the
      pre-commit hook.
- [ ] Per-kata `README.md` exists with objective, walkthrough (baseline vs.
      normalized diff), anti-pattern defense, run instructions, reflection
      — authored during `/iikit-07-implement`.
- [ ] Every non-trivial function, hook, schema, and control-flow branch
      carries a *why* comment tied to an FR-XXX / SC-XXX / anti-pattern —
      enforced during implement review.
- [ ] Reflection note records the observed failure mode the kata prevents
      (context saturation + misread arcane codes); lives in the README's
      reflection section.
