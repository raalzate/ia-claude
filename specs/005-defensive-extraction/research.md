# Phase 0 Research: Defensive Structured Extraction with JSON Schemas

## Decisions

### D-001 — Force extraction via `tool_choice={"type": "any"}`

- **Decision**: Every call to `messages.create(...)` in this kata passes
  `tools=[extraction_tool]` AND `tool_choice={"type": "any"}`. The returned
  response is branched on `response.content[0].type == "tool_use"`; anything
  else raises `ExtractionNotForced` and halts with a labeled reason.
- **Rationale**: FR-002 requires a guarantee that the model returns a tool
  call rather than a free-text "best guess". `tool_choice={"type": "any"}` is
  the Anthropic-documented mechanism for forcing the model to emit *some*
  tool call; since only one extraction tool is registered, "any" collapses to
  "that one". This keeps the boundary between model prose and structured
  output mechanical rather than prompt-engineered.
- **Alternatives considered**:
  - *`tool_choice={"type": "tool", "name": "extract_record"}`.* Equivalent in
    outcome here, but "any" teaches the broader defensive pattern that
    generalizes to multi-tool extraction suites. Documented in the README
    reflection.
  - *Free-text JSON + post-hoc parse.* Rejected: this is the anti-pattern
    FR-002 exists to prevent. Silently accepts a trailing explanation paragraph
    and drops half the record.
  - *`response_format` / JSON mode.* The Anthropic SDK does not expose a JSON
    mode equivalent to OpenAI's; tool-use is the native mechanism. Rejected
    as non-existent rather than inferior.

### D-002 — pydantic v2 as the single source of truth, not hand-written JSON Schema

- **Decision**: `ExtractedRecord`, `SchemaDefinition`, `AmbiguityMarker`, and
  `FabricationMetric` are pydantic v2 models. The tool `input_schema` passed
  to the SDK is generated by `ExtractedRecord.model_json_schema()`. The
  `contracts/*.schema.json` files in this feature dir are the *frozen* output
  of those models, committed for audit / cross-language consumers.
- **Rationale**: Two separately maintained representations (a hand-written
  JSON Schema + a parallel Python type for parsing the response) is exactly
  the drift vector the kata anti-pattern exploits. One source of truth means
  the schema lint (D-003) can walk the *same* object the tool is built from —
  a hand-written `*.schema.json` would not be walkable by Python type
  introspection without re-parsing it, and any gap between the two files
  would go undetected. Constitution Principle II (NN) demands schema
  enforcement at the boundary; pydantic gives us runtime enforcement AND
  a walkable type surface AND JSON Schema generation in one object.
- **Alternatives considered**:
  - *Hand-written JSON Schema + `jsonschema` validator + TypedDict.* Rejected:
    no static type surface for the schema lint; duplicated definitions drift
    silently.
  - *`jsonschema` only (no pydantic).* Rejected: satisfies FR-001/FR-006 on
    paper but leaves FR-004 (the "never declare optional as plain str"
    anti-pattern defense) un-auditable — nothing to introspect.
  - *dataclasses + manual `__post_init__`.* Rejected: no JSON Schema
    generation; would require a parallel schema file, re-opening the drift
    vector.

### D-003 — Schema-lint test walks `ExtractedRecord` fields to defend FR-004

- **Decision**: `tests/katas/005_defensive_extraction/unit/test_schema_lint.py`
  imports `ExtractedRecord`, iterates `model_fields.items()`, and for every
  field with `is_required() is False` asserts the annotated type's union
  includes `NoneType` AND the non-None arm is not a bare `str`. Failure
  message names the offending field and points at FR-004.
- **Rationale**: FR-004 is the kata's primary anti-pattern gate. Encoding it
  as a test that fails the build when someone writes
  `nickname: str = ""` (the classic hallucination magnet) turns the rule
  from a style comment into a CI signal — directly parallel to Kata 1's AST
  lint against `import re` (`specs/001-agentic-loop` D-005). Operationalizes
  Principle II at the pedagogical layer.
- **Alternatives considered**:
  - *Review-only.* Rejected — human drift is exactly what the constitution
    warns against.
  - *`ruff` custom rule.* Deferred; the pydantic introspection test is tighter
    because it runs against the live model object, not the source text.

### D-004 — Fabrication-rate counter over a labeled fixture corpus

- **Decision**: The fixture corpus is six items
  (well-formed, missing-optional, ambiguous-enum, contradictory, empty,
  out-of-enum). Each carries an `expected.json` with two labels: a
  `null_map` listing the optional fields that MUST come back `None`, and an
  `escape_map` listing enumerated fields that MUST route to the escape
  option with a non-empty `details` string. `FabricationMetric.compute(...)`
  walks the extracted record against those labels and increments the
  counter for every optional field that is non-null despite the source
  lacking textual support. SC-001 asserts the counter is 0 across the
  corpus.
- **Rationale**: SC-001 is un-assertable against text identity — the model
  may phrase valid extractions many ways. The only mechanical formulation
  of "zero fabrications" is an audit against a human-labeled null map, one
  the fixture author constructs by reading the source. This also operationalizes
  SC-004 (null rate ≥ 99%) and SC-003 (100% of ambiguous cases escape)
  with the same machinery.
- **Alternatives considered**:
  - *Model-graded audit ("ask another model if this was fabricated").*
    Rejected: pushes the audit back into probabilistic space, directly against
    Principle I. Also un-reproducible in CI.
  - *String containment ("every non-null optional must appear verbatim in the
    source").* Rejected: too brittle (fails on paraphrase, numeric
    normalization, capitalization). The labeled `null_map` is strictly
    stronger — it asks "did the human labeler consider this field absent?".

### D-005 — Recorded fixtures over live API calls in tests, `LIVE_API=1` escape hatch

- **Decision**: Default test run consumes recorded SDK responses from
  `tests/katas/005_defensive_extraction/fixtures/<case>/recorded_response.json`
  via the same injectable client shape introduced by Kata 1. Live calls are
  gated on `LIVE_API=1` and exist only for the runner CLI.
- **Rationale**: FR-006 / SC-002 / SC-003 / SC-004 / SC-001 are all assertions
  over deterministic control flow (schema validation outcome, null presence,
  escape-enum routing, fabrication counter). None of those need a live model
  to exercise; re-rolling the model on each CI run would make the counter
  non-deterministic. Matches the shared baseline declared in
  `specs/001-agentic-loop/plan.md`.
- **Alternatives considered**:
  - *Live calls in CI.* Rejected: flakiness + quota + non-determinism.
  - *VCR.py cassettes.* Rejected as overkill for six fixtures; plain JSON files
    are clearer teaching code.

### D-006 — `extra="forbid"` on `ExtractedRecord` to operationalize FR-010

- **Decision**: `ExtractedRecord` sets `model_config = ConfigDict(extra="forbid")`.
  The response-handling path calls `ExtractedRecord.model_validate(tool_use.input)`;
  any extra key raises `ValidationError` and surfaces to the caller (FR-008).
- **Rationale**: FR-010 requires refusal of fields outside the declared
  schema. `extra="forbid"` is the pydantic-native, mechanical enforcement;
  anything else would be a hand-rolled diff we'd have to maintain.
- **Alternatives considered**:
  - *`extra="ignore"` + post-hoc key-set diff.* Rejected: silently drops
    the very signal FR-010 asks us to surface.

### D-007 — Single project layout; reuse Kata 1 client shape

- **Decision**: Source at `katas/005_defensive_extraction/`; tests at
  `tests/katas/005_defensive_extraction/`. The injectable client wrapper
  mirrors the shape introduced in Kata 1 so both katas share a recorded-
  response test strategy.
- **Rationale**: FDD cadence from Constitution v1.3.0 (Development Workflow)
  wants each kata independently buildable. Matching Kata 1's client shape is
  *convergent*, not coupling — no shared import across katas; the same
  pattern is re-instantiated locally.
- **Alternatives considered**:
  - *Shared `common/client.py` package.* Rejected until a third kata needs it;
    premature consolidation violates YAGNI and blocks per-kata grading.
